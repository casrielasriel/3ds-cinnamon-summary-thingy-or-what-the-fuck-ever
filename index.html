3DS x a look into cinnamon..

A breakdown of what the cinnaamon project, and the planned cinnamon Launcher, can do on real 3DS hardware. Hardware ceilings, optimization impact, and per game compatibility.

What the percentages mean

100% = the game holds a steady 30 FPS on that hardware for the entire runtime. No dropped frames, no hitches. This does not include the occasional microstutter because of i/o access stuff.

Below 100% = the game cannot sustain 30 FPS in every frame. The number is the rough fraction of frames that complete within the 33ms budget. A game at 96% drops roughly 1 frame per second. At 90% it drops 3 frames per second.

Single-frame hitches happen when a frame takes 50-80ms. You notice a microstutter.. the game hangs for a split second and then resumes. These may come from surface chain transitions (allocating FCRAM + display transfer), texture upload bursts from SD card, or large with() loops that generate too many VM ops in one frame.

Sustained drops happen when the average frame time exceeds 33ms. The game will feel slower and the audio will desync. These can come from rooms that pushes too many draw calls, too many layers of surface compositing, or saturated VRAM spilling into FCRAM access.

Texture load stutter comes from PNG decompression and GPU upload on the ARM11 (confirmed this on my real o3DS with the old cinnamon v0.20 beta). Baked tex3ds textures eliminate it entirely.. the PICA200 reads precompressed data directly from FCRAM with zero CPU decode. Why did we do PNGS originally..????

    cinnamon Launcher (still in development)

    cinnamon Launcher is a work in progress GameMaker runner for the 3DS, and runs any game that doesnt have a port on 3DS already or if you just want to play that game without all the bottom screen/touch stuff. Basically, its one app on the 3DS that launches any compatible GameMaker game from a single library, not a separate build per game. Preprocess your data.win, put the files wherever i decide the files to go when i start working on it, and the launcher runs the game.

    My research so far points to the majority of GameMaker bytecode games running at 30 FPS on O3DS, old hardware included... but with pretty much one exception category: games doing heavy true 3D rendering. No, Deltarune does not fall into that exception. Its d3d_* calls are compatibility shims; the only one actually used is d3d_set_fog for sprite glow and flash effects, which maps to PICA200 hardware fog at basically zero cost. No ACTUAL 3D geometry runs anywhere in Deltarune, even if it seems like it does in Chapter 5.

    Bytecode compatibility is not the same as a working game. Cinnamon implements GML functions one at a time. A game can use a fully supported bytecode version and still call a function Cinnamon has not implemented yet. That function needs to be written before the game runs correctly... but sometimes, a game can still run fine without that function. This includes window functions (such as when Chara moves your window when they kill you, because obviously, the 3DS does not run windows and therefore it doesnt need to move the window around, because there is no windows.) This is an open implementation gap that closes over time, not a dead end like YYC or GMRT (they cant even run on 3ds anyhow.)

    What does this assume?

    This document assumes the following code infrastructure exists and works.

    • Cinnamon. A 3DS and Wii U bytecode interpreter for GameMaker Studio games. It is a fork of MrPowerGamerBR's Butterscotch, adding 3DS and Wii U platform backends to Butterscotch's existing Windows, Web, PS2, and PS3 targets. It reads data.win and runs the game loop via a fetch-decode-dispatch loop. It also supports code overrides where individual GML functions can be replaced with hand-written C for hot paths (such as Asriels background and the Hotland lava.) Performance is between the GameMaker Studio runner and a native compiled build (YYC or GMRT).
    • citro3d renderer backend. A full implementation of the RendererVtable using citro3d for direct PICA200 control. This gives manual vertex batching (grouping sprites by texture into one C3D_DrawArrays call), FCRAM textures read with no VRAM copy needed, surfaces that render into VRAM and then copy to FCRAM right after, and no citro2d dependency. citro2d does its own internal GPU command batching but submits one draw per sprite call, which is the key bottleneck this avoids. citro2d sucks. It REALLY does.
    • File system backend. SD card and romfs access via libctru. Asset loading uses pre baked formats, not runtime PNG decode.
    • Audio backend. BCWAV playback through the CEVA DSP via ndsp. OGG decode on ARM11 is avoided.
    • Shader preprocessing pipeline. GML shaders are translated to PICA200 texenv configurations offline. No GLSL reaches the runtime. Not CURRENTLY in cinnamon but for DELTARUNE, it'll be added.
    • Asset baking pipeline. All textures are converted to RGB5551, packed into atlas pages via tex3ds, and stored as t3x files. All audio is converted to BCWAV. All tilemaps are baked to single textures where possible.
    • Input backend. HID input mapped to GML keyboard and gamepad constants. Touch screen will be mapped to mouse in the future.

    Without these systems, the numbers below do not apply. With PNG and OGG at runtime, expect visible stutter on every room transition. To sum it up though.. it'll run really bad. This is what we did originally and it was HORRIBLE.

    The bottom screen is 320 x 240 pixels. In 3D mode.. the top screen renders 400 x 240 per eye, requiring two framebuffers of 384 KB each (768 KB total for both eyes). In 2D mode the top screen uses a single 384 KB framebuffer plus a display buffer.

    FCRAM bandwidth is 3.2 GB/s (3dbrew). The ARM11 runs a single instruction-issue ARMv6K core with Thumb2 support. It has 32 KB L1 instruction cache, 32 KB L1 data cache, and 256 KB L2 cache on O3DS. N3DS adds 2 MB L2 cache and quadruples the clock speed to 804 MHz. The L2 cache difference is significant for with() loops that traverse hundreds of instances. O3DS evicts L2 frequently during instance sweeps, forcing main FCRAM fetches at 3.2 GB/s throughput. N3DS keeps the working set hot in its 2 MB L2, cutting FCRAM hits dramatically. The ARM11 has no branch prediction or out-of-order execution; every cache miss stalls the pipeline until data arrives. Tight inner loops (like vertex batching in citro3d) benefit enormously from L2 residency on N3DS.

    The PICA200 uses DMP's MAESTRO 2G architecture. It has 4 programmable vertex processors. One can act as a geometry engine for subdivision. The fragment pipeline is fixed function with 5 TEV stages (DMP spec sheet, Wikipedia, HandWiki). Some secondary sources cite 6 but the DMP and Wikipedia documentation consistently says 5. There are 3 active texture units. The fourth is reserved for procedural texturing and is rarely used. The PICA200 does not support geometry shaders or tessellation. All vertex and fragment work happens in fixed function pipelines or via the TEV stages. The vertex processors do per-vertex math: transform, lighting, and vertex attribute calculations. The TEV stages do per-fragment math: texture blending, color operations, and output merge. No programmable logic exists between fragments.

    Supported MAESTRO extensions include per pixel lighting, shadow mapping, procedural textures, gaseous object rendering, polygon subdivision, cel shading, volumetric fog, and bump mapping. The 5 stage TEV can replicate most common fragment operations. No programmable pixel shaders are available; every GML shader must convert to TEV configurations.

    Theoretical limits at 268 MHz are roughly 20.5 million triangles per second for geometry and 1,072 million pixels per second for fill. At 200 MHz the DMP spec sheet rates are 15.3 million triangles and 800 million pixels. The 268 MHz figures are the 200 MHz spec scaled linearly by 268/200; real world scaling is never perfectly linear so treat these as an approximate ceiling, not a hard specification.

    The O3DS is CPU limited. The ARM11 at 268 MHz cannot feed the PICA200 draw calls fast enough. Commercial 3DS games typically push 3 to 6 million triangles per second. The N3DS at 804 MHz with 2 MB L2 cache solves this bottleneck entirely.

    APT_SetAppCpuTimeLimit(30)

    The 3DS kernel caps the application thread's CPU time to 30 percent of the ARM11 core via APT_SetAppCpuTimeLimit(30). This is not a suggestion; the kernel preempts the application thread when it exceeds the cap and yields to GSP and other system threads. On O3DS at 268 MHz, 30 percent of one core is approximately 80 MHz effective or 10 ms per 33 ms frame. The remaining 23 ms is reserved for system services (GSPGPU command processing, HID polling, filesystem I/O, network, and idle loops).

    This cap means any frame that spends more than 10 ms of CPU time will starve GSP. GSP handles GPU command queue submission and PDC display transfers. When GSP does not get enough CPU time, the GX command queue backs up (15 entries max per client). A full GX queue stalls C3D_FrameEnd waiting for GSP to drain it. The stall appears as a dropped frame despite the application thread having CPU time remaining. Crash dumps #02 and #03 (undefined instruction in linear heap, GX queue overflow root cause) are the extreme outcome of this starvation pattern.

    The cap is non-negotiable. APT_SetAppCpuTimeLimit accepts values from 10 to 99 percent. Values above 30 cause GSP starvation on O3DS, producing the same symptom: dropped frames from GX queue backup. N3DS at 804 MHz has 30 percent = ~240 MHz effective or 30 ms per 33 ms frame, which is the full frame budget. This is why games that hit 98 percent on O3DS hit 100 percent on N3DS despite identical GPU: the N3DS is never starved for CPU to feed GSP.

    Practical effect for Cinnamon: the interpreter loop, draw call submission, surface DMA setup, and texture upload all compete for the same 10 ms budget. Any single operation that exceeds 10 ms will push the frame over budget. With citro3d batching reducing draw call submission to 1-2 ms and BCWAV audio eliminating OGG decode (3-5 ms), the interpreter loop becomes the dominant term, typically 3-8 ms on a complex Deltarune room. Total CPU time: 5-11 ms, fitting within the cap on most frames. The occasional 12-13 ms spike (rare with() loop over 500 instances, texture upload on room transition) causes the dropped frame that brings 98 percent games down 2 percent.

    SD card throughput

    The O3DS SD controller connects via the GPIO bus, not the MMC controller used by the game card slot. Measured throughput on genuine hardware:

    The GPIO bus limit of ~5 MB/s means a data.win file of 30 MB takes approximately 6 seconds to parse on cold boot. Texture atlas pages for a Deltarune chapter (typically 8-12 MB aggregate) take 2-3 seconds to load on room transition. This is the dominant startup cost and the source of visible loading pauses between rooms. Async SD reads via the filesystem backend can overlap with gameplay for texture group streaming, but the data.win parse itself is synchronous and cannot be deferred: the VM needs the CODE, FUNC, and STRG chunks before any script can execute.

    Sequential reads are 3-4x faster than random reads. All romfs assets should be laid out contiguously and accessed sequentially. The tex3ds and BCWAV conversion output is a small number of large files (atlas pages of 512K-2 MB each, BCWAV tracks of 100K-1 MB each), which maps well to sequential access. The data.win file should remain as a single contiguous file for the same reason.

    The N3DS is a 2014 revision with the same display, GPU, and DSP as the O3DS, but triple the ARM11 clock speed and 2 MB L2 cache instead of 256 KB. The FCRAM doubled from 128 MB to 256 MB, giving ~93 MB usable by homebrew instead of ~31 MB on O3DS. PICA200 and VRAM (6 MB) are unchanged; there is no GPU acceleration gain from the N3DS. This IS a huge upgrade for heavier GML games (such as Pizza Tower as that game uses a heavy particle system. The CPU upgrade helps a LOT.)

    The ARM11 upgrade from 268 MHz to 804 MHz is a 3x frequency increase on the same single-issue in-order core. No branch prediction, no out-of-order execution. The L2 cache increased 8x from 256 KB to 2 MB. This solves the O3DS bottleneck entirely: tight loops (instance sweeps, vertex batching, draw call submission) stay in L2 cache instead of evicting. Cinnamon's bytecode interpreter executes roughly 3x faster on N3DS due to CPU clock alone. The improved L2 residency on texture-bound code and GML loops with good spatial locality gives another 10-20 percent boost beyond clock speed scaling.

    The N3DS has 4 ARM11 cores total. For compatibility, Cinnamon runs on core 0, same as O3DS. The other 3 cores can be used for background work (audio streaming, texture preprocessing, file I/O) if explicitly assigned, but the standard approach is to keep them idle and reserve them for system threads. The second core is not available to homebrew; it runs GSP and system services, same as O3DS. Cores 2 and 3 are usable by homebrew for thread work if explicitly initialized, but game threads are typically single-threaded (core 0 runs game logic and rendering, DSP handles audio).

    FCRAM doubling from 128 MB to 256 MB is a major practical upgrade for texture streaming and asset loading. A room transition that loads 12 MB of atlas pages on O3DS (taking 2-3 seconds over SD card at 5 MB/s) can be interleaved with gameplay on N3DS because there is 93 MB of heap space available instead of 31 MB. More textures can stay resident without having to unload and reload between rooms. The extra FCRAM also helps surface compositing chains; intermediate surface buffers can stay in FCRAM longer without competing with texture data for the same 31 MB pool.

    GPU throughput is identical to O3DS. The PICA200 at 268 MHz is not the bottleneck on either system. Games that run at 98 percent on O3DS (one frame per second below 30 FPS target) jump to 100 percent on N3DS because the CPU can now feed draw commands fast enough. Games that run at 50 percent on O3DS (losing 15 frames per second) jump to 70-80 percent on N3DS depending on whether the remaining bottleneck is fillrate or texture bandwidth. Games hitting the fillrate or texture upload ceiling on O3DS hit the exact same ceilings on N3DS at the same performance level; extra CPU speed does not help those.

    Measured Hardware Ceilings

    I ran a stress test on emulated O3DS hardware. Each axis uses a binary-search probe through escalating load levels. At each level the test renders continuously for a 3-second measurement window. The last level that sustains 30 FPS (within a 1.5 FPS margin, i.e. 28.5 FPS) is recorded as the ceiling. The test then moves to the next level. This continues until the level fails, at which point the previous passing level is the reported ceiling.

    • Geometry. 40,000 quads per frame (80,000 triangles). Tested at {50, 100, 200, 500, 1000, 2000, 5000, 10000, 15000, 20000, 30000, 40000} quads. Each quad is 16x16 px, packed in a grid, single texture. All 40K levels passed; 40K is not a failing boundary, just where the test stopped. The true geometry ceiling is higher. A Deltarune room uses maybe 500 to 1,000 sprite draws; less than 3 percent of this ceiling.
    • Fill rate. 200 full-screen overdraw layers per frame (19.2M pixels/frame at 400x240). Tested at {1, 2, 3, 4, 5, 8, 10, 15, 20, 30, 40, 50, 75, 100, 150, 200} layers. Each layer is a full-screen textured quad. 200 layers passed, the next level was not tested. Pixel art games have maybe 5 to 15 layers; less than 8 percent of this ceiling.
    • Texture binds. 512 unique binds per frame. Tested at {2, 4, 8, 16, 32, 64, 128, 192, 256, 320, 384, 448, 512} 8x8 RGBA8 textures. Each bind is followed by one C3D_DrawArrays call. 512 passed; the test stopped because 512/frame is the practical batch cap (PICA200 hardware limit ~640). A typical room uses 50 to 200 unique textures. With atlasing, 10 to 30 atlas pages. Well within budget.
    • Surface DMA. 100 transfers per frame at half-res (200x120) RGBA5551 (48 KB per transfer, 4.8 MB per frame, 144 MB/s sustained bandwidth). Tested at {1, 2, 3, 4, 5, 8, 10, 15, 20, 30, 50, 75, 100, 150, 200, 300, 500} transfers. 100 passed; 150 dropped below 30 FPS. Each transfer uses gspSubmitGxCommand plus gspWaitForPPF with a stale-event drain before the main loop. This is the only axis where heavy games get close. With VRAM surfaces the display transfer is bypassed entirely; the copy from VRAM to FCRAM after rendering each surface uses the same DMA engine but is amortized over far fewer transfers per frame.
    • VM throughput. 500,000 raw C integer ops per frame. Tested at {1000, 5000, 10000, 50000, 100000, 500000, 1000000, 5000000, 10000000, 50000000, 100000000, 500000000} iterations of d += (d ^ i) * 3 per frame. 500K passed; 1M dropped below 30 FPS. This is a synthetic proxy for ARM11 ALU throughput, not Cinnamon's actual bytecode dispatch speed. Cinnamon's interpreter manages roughly 50,000 to 100,000 bytecode ops per frame on O3DS depending on opcode mix and L2 residency. Chapter 5 rooms with complex with() loops need maybe 5,000 to 15,000 ops. The divergence between raw ALU throughput and bytecode dispatch is caused by the fetch-decode-dispatch overhead (memory reads for opcode bytes, computed-goto target resolution, operand extraction) that the synthetic test does not model.
    • Combined stress. 5x simultaneous scale factor. The combined test runs all axes concurrently: 2,500 quads (geometry), 500 tex binds (capped), 50 surface DMAs, 250,000 VM iters per frame. Tested at scale factors {0.1, 0.25, 0.5, 1.0, 1.5, 2.0, 3.0, 5.0, 8.0, 10.0, 15.0, 20.0, 30.0, 50.0, 100.0}. Scale 5.0 passed; scale 8.0 dropped below 30 FPS. At scale 5.0 the test represents 5x the per-frame workload of a typical heavy Deltarune room across all axes simultaneously, confirming no hidden interaction between axes causes early failure.

    FCRAM texture validation

    FCRAM texture sampling was validated on genuine and real O3DS hardware. 10 textures of 512x512 RGBA8 (10 MB total) were bound as PICA200 texture sources from FCRAM with no VRAM copy. The test rendered a full-screen quad for each texture at 60 FPS target. Result: 59.8 FPS sustained across all 10 textures. The FCRAM texture path on PICA200 works correctly and incurs no measurable performance penalty compared to VRAM-sourced textures at this data volume. This confirms that texture atlas pages stored in FCRAM can be sampled directly by the GPU without the VRAM upload step required by citro2d.

    Optimization Stack

    Every optimization I can apply, ordered by impact. These are the systems that turn an unplayable port into a smooth one.

    Critical

    • Critical Cinnamon bytecode interpreter
      Replaces the GameMaker Studio runner's interpreter with a more efficient one. Optimized dispatch loop, cached function lookups, and code overrides for hot paths. Cinnamon runs roughly 2 to 5 times faster than the original interpreter on bytecode-heavy rooms.
    • Critical citro3d with vertex batching
      Groups sprites by texture. Submits one draw call per texture instead of one per sprite. No redundant state validation per draw. Reduces CPU draw call overhead by 10 to 50 times.
    • Critical VRAM surface render targets
      citro3d renders each surface into VRAM, then copies the finished result into FCRAM right after the draw. The VRAM slot frees immediately instead of staying occupied for the rest of the frame. This is the planned fix for the Chapter 2 and beyond surface bottleneck: VRAM no longer has to hold every active surface at once, only one at a time during the render-then-copy step. Zero PDC display transfer cost on the composite step itself. Without this, Chapter 5 would spend half the frame budget on surface DMA.
    • Critical RGB5A1 and RGBA4444 preprocessing
      Cuts texture size by 2 to 4 times. Less memory bandwidth. More room for atlas pages in FCRAM. Reduces texture upload time from SD card.
    • Critical Texture atlasing (tex3ds)
      Puts 50 or more sprites into one atlas page. One draw call for 50 sprites. tex3ds also handles swizzling (PICA200 tile format) and mipmap generation. The output .t3x file is directly uploadable to the GPU with zero preprocessing at runtime. Directly addresses the O3DS ARM11 draw call submission bottleneck.
    • Critical Pre compiled shaders
      GML shaders converted to PICA200 texenv configurations offline. Zero runtime GLSL translation. TEV executes fixed function operations at no GPU cost.
    • Critical FCRAM textures
      PICA200 samples textures directly from FCRAM. No VRAM copy on bind. Proven at 59.8 FPS with 10 MB of textures on real hardware.
    • Critical BCWAV audio preprocessing
      CEVA DSP handles hardware decoded audio. Near zero ARM11 CPU cost. OGG decode on ARM11 eats up to 30 percent of the frame budget. This eliminates that.
    • Critical PNG to raw texture pre baking
      Eliminates 5 to 15 millisecond PNG decode per sprite sheet at load time. Output is swizzled .t3x directly readable by the PICA200. Optionally ETC1 compressed (4:1 over RGBA8) for atlas pages at a quality tradeoff: block artifacts on gradients, acceptable on pixel art with clean color regions. Room transitions load 200+ milliseconds faster. No visible stutter.

    High

    • High Half resolution surfaces (200 x 120)
      4 times fewer pixels per surface. RGB5A1 halves it again. Total 8 times reduction versus full resolution RGBA8. VRAM usage per surface drops from 384 KB to 48 KB.
    • High Sprite culling
      Removes off screen sprites from the draw list. Busy rooms see 30 to 50 percent fewer draw calls. Simple frustum check per instance.
    • High Tilemap baking
      Pre renders static background tiles to a single texture per layer. Removes 200 or more individual tile draws per room. Huge win for rooms with complex tile based backgrounds.
    • High Culling based depth sorting
      Sort draw list by depth and texture once per frame. Skip invisible instances before entering the render loop. Reduces CPU sorting overhead.
    • High Static UI text and HUD baking
      Render HUD elements that rarely change to a cached texture. Only redraw when values update. Saves 10 to 20 draw calls per frame.

    Medium

    • Medium Remove application surface
      Draw directly to the screen framebuffer. Saves one full 400 x 240 blit per frame, roughly 1 to 2 milliseconds.
    • Medium Dynamic resolution scaling
      Drop to 320 x 192 during heavy frames. 36 percent fewer pixels to fill. PICA200 display transfer scales to full resolution at nearly no cost. Switch back to full res when the room calms down.
    • Medium DSP hardware audio mixing
      All mixing runs on the CEVA DSP via ndsp. BCWAV playback costs effectively zero ARM11 CPU time. Frees up CPU for rendering and game logic.
    • Medium Pre computed vertex buffers for static geometry
      Rooms with static backgrounds and platforms get their geometry baked into persistent vertex buffers. No per frame vertex upload needed for background tiles.
    • Medium Texture group streaming
      Organize textures into groups per room. Load the next room's textures during gameplay via async SD reads. Hides load time behind gameplay.

    Cheap wins

    • Cheap TEV palette swap fallback
      Some GML palette effects translate to TEV combiner stages at zero GPU cost. No shader needed.
    • Cheap Pre scaled sprite atlases
      Include 2x scaled versions of sprite atlases for rooms that need them. Avoids runtime bilinear scale on the GPU.
    • Cheap Skip frames on loading
      If a room transition takes more than 100ms, freeze the GPU but keep the DSP running. Audio continues smoothly. Resume rendering when loading completes.
    • Cheap Use PICA200 hardware fog for glow effects
      Many Deltarune glow and flash effects use gpu_set_fog. PICA200 has hardware fog support. This maps directly at no cost.

    FMOD, and why its evil

    Pizza Tower does not store audio in data.win. All sound is in external FMOD Studio bank files: master.bank, sfx.bank, and music.bank, stored alongside the executable. The game uses a custom extension DLL, fmod-gamemaker.dll, built by the Pizza Tower developers specifically for Windows. This DLL bridges GML extension function calls to the FMOD Studio runtime. FMOD Studio itself is a separate C++ middleware library that ships as platform-specific native binaries.

    None of this exists on 3DS and none of it can. A DLL is a Windows PE binary. The FMOD Studio runtime has no 3DS ARM11 build and Firelight Technologies does not publish one. The GML calls in Pizza Tower's code hit functions like fmod_event_create, fmod_event_play, and fmod_param_set. When those functions are called, control leaves GML entirely and enters the native extension. On 3DS there is no native extension to enter.

    The only working approach seen in the wild is the one used by the unofficial Switch port (TurtleHouse, D-Official). Every FMOD function is stubbed in GML. A custom fmod_sound struct type replaces FMOD events. Each event is manually mapped: the bank files are cracked open with FMOD Bank Tools, individual audio files are extracted, and every event's sound list is hardcoded by hand in GML. The music tracks use GameMaker's native audio_play_sound with loop point logic to replicate FMOD's intro-then-loop pattern. Positional sounds map to GameMaker audio emitters. This is a total rewrite of Pizza Tower's audio layer, and I REALLY would prefer not to do this.

    For a 3DS port this work is the same, with the additional constraint that the output audio must be BCWAV rather than OGG or WAV. Every extracted sound file would need conversion through vgmstream or a similar tool before the DSP can play it natively. The fmod_sound stub layer would call ndsp directly or go through a thin GML wrapper over the BCWAV playback backend. The GML-facing API stays identical to what the game expects; the implementation underneath replaces everything.

    This is separately scoped work from the renderer and bytecode work. Rendering and logic compatibility does not imply audio compatibility. Pizza Tower needs both.

    Bottleneck Hierarchy

    On O3DS with the full Cinnamon stack, the bottlenecks ordered by severity:

    First. CPU draw call submission. The ARM11 can handle roughly 150 to 200 C3D_DrawArrays calls per frame. With atlasing and batching, a typical room uses 30 to 80 calls. That fits comfortably.

    Second. VRAM exhaustion. Under citro2d, 6 MB of VRAM supports 5 to 6 active surface render targets at once. A typical room uses 1 to 3. Chapter 4 and 5 occasionally need 4 to 5 simultaneously. This was the biggest hurdle for Chapter 2 and beyond. The citro3d render-then-copy approach removes this constraint: each surface draws into VRAM, then citro3d moves the result into FCRAM right after, freeing the slot before the next surface needs it. Surfaces stop competing for a shared pool of simultaneous VRAM slots. This is the planned fix, not yet validated on real hardware the way the FCRAM texture test was.

    Third. FCRAM texture upload bandwidth. SD card reads at roughly 5 MB per second on O3DS (measured, limited by the SD controller on the GPIO bus, not the card itself). A room transition might load 12 MB of atlas pages. That is a 2 to 3 second window where the game is loading. The PICA200 samples textures from FCRAM directly so no VRAM upload cost, but the data must still reach FCRAM from the SD card. Texture group streaming (loading the next room's assets during gameplay) can hide this behind gameplay.

    Fourth. Fragment fillrate. The PICA200 can do 11,000 or more full screen passes per frame at 30 FPS. Never a bottleneck for 2D pixel games.

    The N3DS eliminates the first bottleneck with 3 times CPU speed and 2 MB of L2 cache. Games that hit 98 percent on O3DS hit 100 percent on N3DS.

    Optimization Impact Summary

    The full Cinnamon stack gains roughly 70 to 80 percentage points over the raw configuration across all games.

    The single biggest win is citro3d with vertex batching and atlasing. It reduces draw calls by 10 to 50 times. The O3DS ARM11 cannot submit individual sprite draws at any reasonable framerate. It can submit 30 to 80 batches per frame. That single change is the difference between 18 percent and 50 percent on Chapter 5.

    The second biggest win is the Cinnamon bytecode interpreter. Cinnamon's optimized dispatch loop runs roughly 2 to 5 times faster than the GameMaker Studio runner's interpreter on complex GML. Chapter 4 and 5 loops that iterate 200 instances and call methods on each one benefit significantly from the reduced per-opcode overhead.

    The third biggest win is VRAM surface render targets. They eliminate all PDC display transfer costs per surface. Without this, Chapter 5 would spend half the frame budget or more on surface DMA transfers. With VRAM targets, that cost is zero.

    Key Takeaways

    The O3DS is CPU limited. Every optimization that reduces CPU work translates directly to FPS. Batching, culling, the optimized interpreter, and precomputed draw lists are the correct targets.

    The N3DS fixes the CPU problem. A game at 92 percent on O3DS runs at 100 percent on N3DS. The extra memory and L2 cache also help with texture streaming.

    GML shaders using shader_set are dead on PICA200. There are no programmable pixel shaders. Every GML shader must convert to the 5 stage TEV fixed function pipeline. Most GM effects like color blend, additive, multiply, fog, and palette swap map cleanly. Complex displacement shaders and screen warp effects need pre baking or cannot be reproduced.

    citro2d was the wrong renderer for GameMaker games. It batches GPU commands internally but still submits one draw call per sprite, and it leaves surfaces stubbed out entirely in the reference codebase. citro3d fixes both: manual batching cuts draw calls 10 to 50 times, and FCRAM texture sampling needs no VRAM copy at all.

    VRAM surface exhaustion stops being the constraint once citro3d's render-then-copy approach lands. Surfaces no longer need a permanent VRAM slot for as long as they stay active. They render into VRAM, copy out to FCRAM, and free the slot immediately. Chapter 2 and beyond were the hardest rooms specifically because of simultaneous surface count against a fixed 6 MB VRAM budget. That constraint goes away once this is implemented and validated.

    YYC and GMRT games are a hard wall. Both compile to native code instead of bytecode. No bytecode means Cinnamon has no input to read. The only path forward is a full manual reimplementation of the game logic in C, which defeats the purpose of using Cinnamon.

    Preprocessing is mandatory. PNG decode at runtime causes visible stutter on every room transition. OGG decode on ARM11 eats 15 to 30 percent of the frame budget. GLSL at runtime needs CPU based translation into TEV configurations. All of these must be done ahead of time.

    The PICA200 is not the problem. It is a 268 MHz GPU designed for a 2011 handheld that ran Monster Hunter 4 Ultimate, Kid Icarus Uprising, and Luigi's Mansion 2. Pixel art GML games are far below its design target. The only enemy is the ARM11 core, and every optimization in the stack is aimed directly at it.

    Technical Reference

    Memory Map

    Key memory regions on O3DS hardware (3dbrew):

    • VRAM. 0x18000000 to 0x18600000. 6 MB total. On SoC graphics memory. Integrated with the PICA200 GPU on the same die. Fast path for GPU reads. Framebuffers, render targets, and surface objects must live here for GPU access. Texture bind from FCRAM works without a VRAM copy but the texture object metadata goes through VRAM. VRAM access from ARM11 is slow (requires GSP cooperation); avoid reading from VRAM on the CPU side. VRAM bandwidth for GPU reads is high; GPU writes during rendering are not a bottleneck.
    • FCRAM. 0x20000000 to 0x28000000. 128 MB total (256 MB on N3DS). System RAM. ~31 MB usable by the application on O3DS after OS reservation, ~93 MB on N3DS. PICA200 can sample textures from FCRAM directly at full speed (3.2 GB/s), no VRAM copy required. The ARM11 runs code from FCRAM with L1/L2 cache. FCRAM is on a separate bus from VRAM. FCRAM access from the ARM11 fills the L1 and L2 caches; repeated access to the same address from within a loop stays hot. FCRAM access from the PICA200 bypasses caches but is memory-bandwidth limited by the 3.2 GB/s bus. GPU and ARM11 contend on the FCRAM bus; high ARM11 traffic (e.g., large with() loops) can reduce effective PICA200 FCRAM throughput.
    • DSP RAM. 0x1FF00000 to 0x1FF80000. 512 KB dedicated RAM for the CEVA TeakLite DSP. Audio buffers and DSP firmware live here. ARM11 can write to it but should avoid reading from it (slow). DSP RAM is not accessible to the PICA200.
    • Shared memory. GSP shared memory at 0x1FF80000. Configuration memory at 0x1FFF8000. IO registers mapped from 0x10100000 upward. GSP shared memory holds the GX command queue, frame sync events, and GPU status. ARM11 and GPU synchronize via shared memory events.
    • N3DS additional memory. 0x1F000000 to 0x1F400000. 4 MB block. Used by QTM (face tracking) services. Only the first 2 MB is GPU accessible. Often counted in the 10 MB VRAM figure from Digital Foundry and Wikipedia, though it is technically separate from VRAM.

    CPU Time Limit

    The Cinnamon reference codebase calls APT_SetAppCpuTimeLimit(30). On O3DS, the ARM11 runs the application thread and the GSP (GPU service) thread on the same core. This setting caps the application thread to 30 percent of CPU time. The remaining 70 percent is reserved for GSP and system threads. The application effectively has about 80 MHz of the O3DS ARM11 for game logic and draw call submission. citro3d command processing happens on the GSP side and is not counted against this limit.

    The ARM11 is a single-issue in-order processor. No branch prediction, no out-of-order execution. Instruction throughput is one per cycle on ideal code with no stalls. Cache misses block the pipeline until data arrives from FCRAM (roughly 10-15 cycles for a miss). Tight loops with good L1/L2 locality run at full 268 MHz throughput. Loops that miss L2 every iteration effectively run at 268 MHz / 15 = ~18 MHz due to stalls. This is why the Cinnamon interpreter's opcode caching and code override system for hot paths exist.

    The ARM11 executes ARMv6K ISA. Thumb2 is supported for 16-bit instruction encoding. ARM instructions are 32-bit. Thumb2 is preferred for code density and often for performance on in-order processors; smaller code footprint means better L1 locality. Game logic (the VM interpreter loop) typically compiles to Thumb2.

    A game running at 30 FPS has 33ms per frame. With a 30 percent CPU limit, the application gets roughly 10ms of CPU time per frame. Every millisecond of CPU work saved by batching, culling, and the optimized interpreter is directly measurable.

    PICA200 Graphics Pipeline

    The PICA200 pipeline follows this sequence: vertex fetch from vertex buffers, vertex shader execution (fixed function for standard transforms or custom via vertex constants), primitive assembly (triangles, lines, points), viewport transform, rasterization, fragment shader execution (fixed function TEV stages), output merge (color write, depth test, stencil test), and framebuffer write.

    Vertex processing happens in 4 parallel vertex processors. Each processes one vertex per cycle at 268 MHz. Vertex constants (44 vec4 registers per processor) hold transformation matrices, lighting parameters, and user-defined constants. Custom vertex shaders are not supported; vertex work must use fixed function transforms or be pre baked (e.g., vertex buffers are already transformed on the CPU).

    The rasterizer converts triangles to fragments at the PICA200's fill rate: 1,072 million pixels per second at 268 MHz. This is far beyond the needs of any 2D GML game. The fillrate ceiling is not the problem.

    Fragment processing happens in the TEV unit. The TEV (Texture Environment) has 5 stages. Each stage inputs texture results and color, applies combiners (multiply, add, subtract, interpolate, cross product), and feeds the next stage. The final stage outputs the fragment color, which then enters the output merge (color write with blending, depth test, stencil test, alpha test). No programmable fragment shaders; TEV is fully fixed function. Most blend modes (additive, multiply, alpha blend, color blend) map cleanly to TEV stages. Complex fragment effects (bump mapping, parallax mapping, advanced normal mapping) work within TEV limits due to the supported MAESTRO extensions.

    Texture binds happen per draw call. The three active texture units can sample from three separate textures (or the same texture three times with different samplers). Texture sampling is fast from FCRAM (3.2 GB/s) or faster from VRAM. Repeat modes (clamp, wrap, mirror) and filter modes (point, linear) are supported per texture unit.

    The PICA200 handles clipping automatically. Vertices are clipped to the viewport and near/far planes before rasterization. Scissor rects are supported and commonly used to mask rendering to parts of the screen (e.g., dual-screen masking on the 3DS display pipeline).

    PICA200 Texture Formats

    The PICA200 supports these texture formats. Only the efficient hardware-native ones are relevant for this project:

    • RGBA8 (32-bit). 4 bytes per pixel. Full color quality. Never use for textures. Only for framebuffers and display targets.
    • RGB8 (24-bit). 3 bytes per pixel. Not commonly used. No alpha channel.
    • RGBA5551 (16-bit). 5 bits per channel plus 1-bit alpha. 2 bytes per pixel. 32 colors per channel. Good enough for pixel art. Half the bandwidth of RGBA8.
    • RGB565 (16-bit). 5 red, 6 green, 5 blue. 2 bytes per pixel. No alpha. Good for opaque sprites and tiles.
    • RGBA4444 (16-bit). 4 bits per channel. 2 bytes per pixel. Less color depth than RGBA5551 but same size. Use RGBA5551 instead unless alpha precision is not needed.
    • LA8 (16-bit). Luminance 8 + Alpha 8. 2 bytes per pixel. Used for text rendering and monochrome sprites with alpha.
    • LA4 (8-bit). 4 bits luminance, 4 bits alpha. 1 byte per pixel. Good for small UI elements and fonts.
    • L8 (8-bit). 8-bit grayscale. 1 byte per pixel. Masks, glow maps, light maps.
    • A8 (8-bit). 8-bit alpha only. 1 byte per pixel. Alpha masks.
    • ETC1 / ETC1A4 Compressed. 4 bits per pixel (ETC1) or 8 bits per pixel (ETC1A4). PICA200 hardware decodes ETC1. This gives 8x compression over RGBA8. Worth using for large atlas pages. Not widely used in practice because ETC1 has visual artifacts and most GML games ship PNG/RGBA8 assets, but the preprocessing pipeline can convert to ETC1 offline.

    Texture dimensions must be powers of two. The PICA200 supports up to 1024 x 1024. Non-power-of-two textures are padded by the driver. This wastes VRAM. All atlas pages should be power of two aligned (512 x 512 or 1024 x 1024).

    Textures must be swizzled (tiled) for the PICA200. The tex3ds tool handles this automatically. Unswizzled textures will show as garbage or cause GPU hangs.

    GSP/GX Command Queue

    The GSP has a per-client command queue with 15 entries. Each entry is 8 words (32 bytes). The queue is at GSP shared memory base + (clientID * 0x200) + 0x800 header space. The 15-entry limit means at most 15 GPU commands can be queued before the application must wait for GSP to drain some.

    C3D_FrameEnd submits a GX command internally to flush the GPU pipeline. It also triggers the PPF (P3D Frame) event with auto-reset. If the PPF event is consumed by gspWaitForPPF before FrameEnd's completion, it returns immediately with a stale signal. This was the root cause of crash dumps #02 and #03 (undefined instruction at PC 0x14001844 with magic 0xDEADBEEF / 0xDEADC0DE). The fix is to call gspWaitForPPF before submitting GX batches and to use a batch size of 1.

    The GX command format uses 8-word packets. Command ID 0x01000103 is used for display transfers (the PDC unit). Geometry and rendering commands use C3D internally via the GX command buffer. citro3d allocates a 524,288 byte command buffer by default (C3D_Init(0x80000) in the Cinnamon reference codebase).

    Linear Heap

    libctru provides a linear memory allocator (linearAlloc/linearFree) for DMA-accessible memory. On O3DS, the linear heap is 4 MB by default. The extended linear heap can be configured up to 12 MB if needed via osSetLinearHeap or the system config.

    All GPU buffers (vertex buffers, texture data, GX commands) must be allocated from the linear heap. This means user code cannot use more than roughly 4 MB of linear memory on O3DS without explicit extension. The stress test source (source/main.c) uses linearAlloc for vertex buffers and texture pools. The crash dumps pointed to linear heap addresses (PC 0x14001844 is in FCRAM/linear heap range, not VRAM).

    The reference Cinnamon debug monitor tracks linearSpaceFree() and osGetMemRegionFree(MEMREGION_APPLICATION) on the bottom screen.

    Audio DSP Details

    The CEVA TeakLite DSP runs at 134 MHz with 512 KB of dedicated DSP RAM. ndsp provides the ARM11 API for DSP communication. The DSP handles hardware mixing of up to 24 audio channels simultaneously.

    BCWAV is the native 3DS hardware audio format. The CEVA DSP decodes BCWAV directly with no ARM11 CPU cost. Each BCWAV channel plays from a buffer in DSP RAM or FCRAM. The DSP reads audio frames via DMA while the ARM11 processes game logic.

    The reference Cinnamon codebase uses ndsp for BCWAV playback. Audio system init allocates a channel cache (tracked via N3DSAudioSystem_getCacheStats showing cachedSounds/totalSounds/cachedSoundBytes/cacheLimitBytes on the debug monitor). The audio update runs once per frame at update(1.0/30.0).

    OGG Vorbis decode on ARM11 would consume 15 to 30 percent of the frame budget. This is why BCWAV preprocessing is marked critical. No OGG decoder runs in the optimized configuration.

    Cinnamon Reference Codebase Notes

    Specific findings from source analysis of the reference codebase at cinnamon-main/src/n3ds/:

    • VRAM constant. N3DS_TOTAL_VRAM_BYTES is defined as (6u * 1024u * 1024u) = 6,291,456 bytes. Confirms 6 MB VRAM on the source code side.
    • N3DS_FORCE_OLD3DS_MODE. Defined as 0 in n3ds_platform_config.h. No forced O3DS mode. The binary is allowed to detect and use N3DS features when running on New 3DS hardware.
    • Frame pacing. The main loop uses N3DS_beginPacedFrame (Coffee's cycle-based frame pacing). Frame ticks = SYSCLOCK_ARM11 / 30. Coarse guard is ~0.5ms. Prevents drift and uneven frame pacing on real hardware.
    • APT CPU time limit. APT_SetAppCpuTimeLimit(30) limits the application thread to 30 percent of the O3DS core. The codebase calls osSetSpeedupEnable(true) to enable N3DS clock speed increase when available.
    • Renderer. Uses citro2d exclusively. citro3d is linked in CMakeLists.txt but no citro3d function is called in the renderer. All draw calls go through C2D_DrawSprite or C2D_DrawRectSolid. No shaders, no batching, no vertex buffer management.
    • Surfaces. createSurface is fully stubbed. Returns -1. Surface functions are never called. There is no VRAM surface management in the reference codebase.
    • Audio worker. N3DS_STREAM_WORKER_CORE_ID is set to 1. This targets the second ARM11 core. On O3DS this core does not exist for homebrew. The audio worker thread would silently fail or run on the wrong core depending on thread creation flags.
    • Boot log. A 6-line rotating boot log is displayed on the loading screen. Data.win chunks are parsed via a progress callback showing the chunk magic (e.g. "Parsing chunk SPRT").
    • Data.win parsing. The parser loads SOND, AGRP, SPRT, BGND, PATH, SCPT, GLOB, SHDR, FONT, TMLN, OBJT, ROOM, TPAG, CODE, VARI, FUNC, and STRG chunks. TXTR and AUDO are NOT parsed at runtime (preprocessed assets in romfs are used instead).
    • Room borders. The codebase has a themed room border overlay system. Borders are loaded from sdmc:/3ds/cinnamon/gfx/borders/*.t3x or romfs:/gfx/borders/*.t3x. The border is selected by room name prefix (room_gaster, room_fire, room_water, room_tundra, room_ruins, room_castle). A fallback border is drawn via C2D_DrawRectSolid calls.
    • Debug monitor. Shows FPS, render ms, VRAM usage (atlas + direct), atlas page count / limit, free system RAM / total, linear heap free, audio cache stats, and current room name. Toggle with SELECT.

    citro3d Architecture and API

    citro3d is an open source 3D graphics library for the 3DS. It wraps the low level GX and GSP (GPU service) APIs and exposes them as a higher level C interface. citro3d handles GPU command submission, memory management, and synchronization so the application does not have to interact with GSP directly. The reference Cinnamon codebase links citro3d in CMakeLists.txt but uses only citro2d (the 2D sprite wrapper) in actual code. The citro3d fork replaces citro2d entirely with direct citro3d calls.

    citro3d's core abstraction is the frame. Each frame, the application calls C3D_FrameBegin, does rendering via C3D_DrawArrays or similar, then calls C3D_FrameEnd. C3D_FrameEnd submits the accumulated GPU commands to the GX queue, waits for the GPU to finish, and swaps framebuffers. The GPU command buffer is allocated with C3D_Init and has a configurable size (default 512 KB, the reference code uses 524,288 bytes = 0x80000).

    Vertex data flows like this: Application allocates vertex buffers from linear heap via linearAlloc. Each vertex buffer holds interleaved vertex data (positions, colors, texture coordinates). The application fills the buffer with per-vertex data (typically on the CPU after batching similar objects). Then C3D_DrawArrays(prim_type, first, count) is called with a primitive type (GL_TRIANGLES, GL_TRIANGLE_STRIPS), starting index, and vertex count. citro3d submits the GPU command to the GX queue, which the PICA200 fetches and executes. The vertex processors read the vertex buffers and execute vertex shaders.

    Texture binding happens per draw call. C3D_TexBind binds a texture to a texture unit (0, 1, or 2). Textures are C3D_Tex objects allocated from linear heap. The texture holds a pointer to the texture data (which can be in FCRAM or VRAM), the texture dimensions, format, and swizzling metadata. citro3d does not copy FCRAM textures to VRAM; the PICA200 reads directly from the source pointer. This is the key optimization: FCRAM textures are bindable at full speed with zero copy overhead.

    Framebuffers and render targets use C3D_RenderTarget objects. A render target binds a color buffer and optionally a depth buffer. Rendering writes to the render target. C3D_FrameBegin sets up the default framebuffer render target (the display). Custom render targets (surfaces) can be set with C3D_FrameSetTarget and rendered to VRAM, then copied out to FCRAM with a display transfer command (handled by C3D if the target is not the default).

    The TEV (texture environment) is configured per draw via C3D_TexEnv operations. The application calls C3D_TexEnvInit to start fresh TEV configuration, then sets up each of the 5 stages with texture binds, color operations (combiner mode), alpha operations, and constant colors. GML shaders (which are user defined blend modes in GameMaker) are preprocessed offline into TEV configurations and baked into data tables at build time. At runtime, Cinnamon looks up the configuration for the requested shader and applies it via C3D_TexEnv calls before the draw.

    Batching is manual with citro3d. The application is responsible for sorting draw calls (or grouping by texture), collecting geometry with the same texture and state, and submitting one C3D_DrawArrays call instead of many. This is what citro3d does not do for you automatically; citro2d does internal batching but still submits per-sprite. The Cinnamon citro3d renderer implements batching in the RendererVtable: when a sprite with the same texture as the last one is drawn, vertices are added to the same buffer and the draw is deferred until a texture change. This reduces draw calls from 200+ per frame (one per sprite) to 30-80 (one per batch of same-texture sprites), a 5-10x reduction.

    GPU synchronization is handled by citro3d. The application does not block waiting for the GPU; citro3d internally uses GSP events (GspShutdownEvent, GspInterruptEvent, among others) to know when the GPU is ready for the next frame. C3D_FrameEnd returns when it is safe to modify the current framebuffer or start a new frame.

    citro3d is a fairly thin wrapper over the raw GX and GSP APIs. An experienced graphics programmer can call the lower level APIs (GX commands) directly if needed, but citro3d is sufficient for most use cases and insulates the application from GSP timing quirks and GX command queue depth limits.

    Deep Hardware Reference: SoC and Memory Architecture

    The Nintendo 1048 SoC

    The entire compute core of the 3DS lives in a single System on Chip branded as the Nintendo 1048 0H (O3DS) or CPU LGR A (N3DS). It contains the ARM11 MPCore CPU cluster, the ARM9 processor, the PICA200 GPU, the CEVA DSP, and the 6 MB VRAM, all on one die. The ARM9 runs at 134 MHz and handles cryptographic services, filesystem access, and low-level hardware initialization. User-mode homebrew has no direct access to the ARM9; it is isolated behind the kernel. FCRAM and NAND are external to the SoC.

    The O3DS SoC has two ARM11 cores. Core 0 (appcore) runs user applications. Core 1 (syscore) runs system software including the GSP (GPU service process). Homebrew cannot use syscore for application threads; it is reserved. The N3DS SoC has four ARM11 cores. Cores 2 and 3 are available to homebrew applications for explicit background threads. Core 1 still runs system services. In practice, most homebrew and all Cinnamon work runs on core 0 alone; the additional N3DS cores go unused unless the application explicitly creates threads and assigns them.

    The FCRAM is a Fujitsu MB82M8080-07L, a CMOS Fast Cycle RAM with LPDDR interface, accessible in a 64-bit bus. Two dies: MB81EDS516545 (the FCRAM die) and MB82DBS08645. The LPDDR interface gives the bandwidth figure of approximately 3.2 GB/s. The ARM11 and PICA200 share this bus, which means heavy ARM11 memory traffic (large with() loops sweeping instance arrays) reduces effective GPU texture sampling bandwidth. This bus contention is the mechanism behind the "CPU limited" characterization of the O3DS: the ARM11 starves the PICA200 of FCRAM bandwidth as well as consuming draw-call submission time.

    ARM11 MPCore: Pipeline and Cache in Detail

    The ARM11 MPCore implements the ARMv6K ISA. It is a single-issue in-order pipeline with no branch prediction and no out-of-order execution. Every instruction executes in program order, one per cycle when the pipeline is fully fed. On a cache miss the pipeline stalls until the data arrives from L2 or FCRAM; there is no reorder buffer to fill the stall slots with other work. Measured stall penalty is roughly 10 to 15 cycles for an L1 miss that hits L2, and significantly more for an L2 miss that goes to FCRAM. At 268 MHz with the 30 percent APT limit (~80 MHz effective), an L2 miss costs approximately 0.5 to 1 microsecond of stall time per occurrence. A with() loop over 200 instances that misses L2 on each iteration loses roughly 100 to 200 microseconds per frame iteration to stall alone.

    The VFPv2 coprocessor handles IEEE 754 single-precision float. It is not a SIMD unit; each VFP instruction operates on a single float register. Throughput for vectorized math is low. GML uses doubles internally for number values; the runner converts to float for draw operations. The VFP stall behavior differs from the integer pipeline but the in-order execution constraint applies equally.

    Thumb2 is the preferred encoding for code density. A 32-bit ARM instruction and a 16-bit Thumb2 encoding can perform the same operation; the smaller code footprint improves L1 instruction cache hit rate. The compiler (devkitARM GCC or Clang) generates Thumb2 by default for the 3DS target. The Cinnamon interpreter hot loop benefits from Thumb2 because the dispatch table and opcode handler prologs fit in fewer cache lines.

    The N3DS clock multiplier is set per-application via a flag in the application's extended header. When the flag is present, the ARM11 kernel enables the 3x multiplier, raising the CPU to 804 MHz. The GPU, ARM9, and DSP remain at their original clock rates; only the ARM11 scales. The 2 MB L2 cache enables on the same flag. An application without the flag runs at 268 MHz even on N3DS hardware. osSetSpeedupEnable(true) is the libctru call that activates the higher speed from homebrew code.

    SDIO and SD Card Bandwidth

    The SD card interface uses SDIO over the GPIO bus. Measured read throughput on O3DS is approximately 5 MB/s, limited by the SDIO controller, not the card speed class. A class 10 or UHS-I card produces the same measured throughput as a class 4 on 3DS hardware; the controller is the ceiling. Nintendo recommends SD cards up to 32 GB formatted FAT32 with 64 KB allocation units. Cards up to 2 TB have been tested and work, with increasing boot time overhead proportional to card capacity. A 256 GB card adds approximately 7 to 8 seconds to boot time versus no SD card.

    For Cinnamon: a room transition loading 12 MB of atlas pages over this 5 MB/s link takes 2 to 3 seconds. Texture group streaming (loading the next room's assets in the background while gameplay continues) is the correct solution. N3DS FCRAM doubling to 256 MB gives 93 MB of heap for homebrew instead of 31 MB, which means more atlas pages can stay resident across transitions without unloading, directly reducing the per-transition load window.

    Deep Hardware Reference: PICA200 in Detail

    Origin and Architecture

    The PICA200 is a graphics IP core designed by Digital Media Professionals Inc. (DMP), a Japanese fabless semiconductor company founded in 2000. DMP's previous work was the ULTRAY2000, a proof-of-concept MAESTRO GPU presented at SIGGRAPH 2005. The PICA200 announced at SIGGRAPH 2005 and demonstrated with an FPGA prototype (the "Mikage" demo, co-developed with Futuremark) at SIGGRAPH 2006. DMP announced Nintendo's adoption of PICA200 for the 3DS on June 21, 2010. The chip is fabbed at 65 nm and is rated at up to 400 MHz; Nintendo clocks it at 268 MHz in the 3DS.

    PICA200 implements the MAESTRO-2G architecture. MAESTRO stands for DMP's proprietary graphics framework. The key design goal was hardware acceleration of advanced shading effects at low power, targeting embedded devices where programmable GPU pipelines (as found on desktop GPUs) would consume too much die area and power. MAESTRO-2G achieves this by encoding complex shading operations as fixed-function hardware stages rather than general-purpose shader programs. The result is a GPU that cannot run arbitrary shader code but can execute specific complex effects (subsurface scattering, soft shadows, procedural textures) in hardware at essentially zero cost, because those paths are hardwired logic rather than shader instruction streams.

    The PICA200 supports OpenGL ES 1.1 as a baseline API, plus DMP's proprietary extensions. The baseline covers vertex arrays, texture mapping, alpha blending, depth test, stencil test, and basic lighting. The extensions add the MAESTRO feature set. In the 3DS homebrew stack, citro3d does not expose OpenGL ES; it exposes the PICA200's hardware directly in a stateful C API that maps to GX commands.

    Vertex Processors and Primitive Engine

    The PICA200 has four programmable vertex processors. Each accepts a vertex from the vertex buffer, executes a vertex shader program (called a vertex shader but implemented via PICA200's own instruction set, not GLSL), and outputs a transformed vertex. The vertex shader instruction set supports 128 operations including matrix multiply, dot product, min/max, and conditional branches. Vertex constants are 96 vec4 registers (32 per vertex shader, per 3dbrew documentation) holding transform matrices, lighting parameters, and user data. The fourth processor, called the Primitive Engine, can be configured as either a geometry shader (processing primitives rather than individual vertices, supporting Catmull-Clark and Loop subdivision) or as a standard fourth vertex processor. For Cinnamon's 2D sprite rendering, the Primitive Engine runs as a fourth vertex processor; no geometry shaders are used.

    After vertex processing, primitives assemble (triangles, lines, points), pass through viewport transform, then enter the rasterizer. The rasterizer generates fragments at the rated 800 million pixels per second at 200 MHz (approximately 1,072 Mpx/s at the 3DS's 268 MHz, scaling linearly). Scissor testing, depth testing, and stencil testing occur per fragment.

    TEV: The Fixed-Function Fragment Pipeline

    The TEV (Texture Environment) unit is the heart of PICA200's fragment processing. It has 6 stages (3dbrew documents 6; the document you are reading previously stated 5 based on secondary sources, but the primary 3dbrew hardware page is the authoritative count). Each stage has three texture/color inputs, a combiner operation (replace, modulate, add, add-signed, interpolate, subtract, dot3-RGB, dot3-RGBA, multiply-then-add, add-then-multiply), and a scale factor (1x, 2x, 4x). The outputs chain: stage 0's output is available as GPU_PREVIOUS in stage 1, and so on. A separate buffer (GPU_PREVIOUS_BUFFER) can store the result of any stage for use two stages later; writing to the buffer in stage N makes it available in stage N+2. The final TEV stage output feeds the output merger (alpha blend, logic ops, color write mask).

    The TEV replaces programmable pixel shaders entirely. GML's shader_set() calls reference GLSL programs; PICA200 has no GLSL pipeline. Every GML shader must be translated to a TEV configuration at build time. Most blend modes map cleanly: additive blending is (GPU_ADD, src, dst), multiply blend is (GPU_MODULATE, src, dst), color tint is (GPU_MODULATE, sprite_color, tint_color), alpha blend is the standard output merger blend. Complex fragment effects involving dependent texture reads, bump mapping, or screen-space operations require more stages and may require pre-baking. The MAESTRO fog extension (used for Deltarune's d3d_set_fog calls) maps directly to the PICA200 hardware fog unit, which applies linear or exponential distance fog at zero TEV cost.

    Procedural texture generation (proctex) is the fourth texture unit's exclusive function. It generates texture data on-chip from a programmable function rather than reading from memory. This enables palette effects, animated UV patterns, and gradient generation with zero memory bandwidth cost. proctex is rarely used in commercial games and has no GML equivalent; it is available to citro3d users directly via C3D_ProcTex.

    citro2d vs citro3d: The Critical Distinction

    citro2d is a 2D sprite library built on top of citro3d. It provides a simplified API: C2D_DrawSprite, C2D_DrawRectSolid, C2D_DrawTriangle, and so on. citro2d handles its own internal vertex batching; it groups consecutive draw calls into batches internally, submitting a C3D_DrawArrays only when state changes force a flush. The critical limitation: citro2d's batching flushes on every texture change. If 200 sprites across 50 different textures are drawn in a single frame, citro2d submits up to 50 draw calls. More importantly, citro2d was written for general 2D use and does not expose surface management; in the reference Cinnamon codebase, surface_create is stubbed to return -1 because citro2d provides no render-to-texture pathway. Surface compositing effects (dark world lighting, CRT overlay, camera shake blit) are impossible without surfaces.

    citro3d is the underlying library citro2d is built on. It exposes the PICA200 directly: vertex buffers, texture binding, TEV configuration, render targets, and GX command submission. The developer is responsible for batching, vertex format declaration, and state management. This is more work but enables the optimizations that citro2d cannot provide. Specifically: manual batching can group all sprites sharing a texture page into one draw call regardless of interleaved draw order, because the developer controls when to submit and can sort by texture first. FCRAM texture binding via C3D_TexBind reads from system RAM directly with zero VRAM copy; this is what the 10 MB real-hardware test confirmed at 59.8 FPS. Render targets (surfaces) work correctly in citro3d via C3D_RenderTargetCreate, C3D_FrameSetTarget, and the render-then-copy approach that frees VRAM slots immediately after each surface draw. None of these are available through citro2d's API surface.

    The practical consequence for Cinnamon: the reference codebase uses citro2d and cannot implement surfaces or achieve the draw call reduction needed for heavy Deltarune chapters. The fork that adds citro3d replaces citro2d entirely, rebuilds the renderer around manual vertex buffers and explicit batching, and implements the VRAM render-then-copy surface system. That fork is what the performance numbers in this document reflect when cited as "optimized." The reference codebase numbers are what appear in the "Cinnamon Current State" chart.

    Deep GameMaker Research

    The data.win File Format

    GameMaker compiles all game assets into a single binary file called data.win on Windows, data.ios on iOS, data.droid on Android, and game.unx on Linux. The format is an IFF-derived container; YoYo Games CEO Russell Kay called it a WAD file, which is the project's internal name. The root container chunk is FORM, and it contains a sequence of named child chunks, each with a 4-byte magic, a 4-byte byte count, and then the chunk data. The complete chunk list in a typical GMS 1.4 build is: GEN8 (general metadata), OPTN (game options), LANG (localization), EXTN (extensions), SOND (sound definitions), AGRP (audio groups), SPRT (sprites), BGND (backgrounds), PATH (paths), SCPT (scripts), GLOB (globals), SHDR (shaders), FONT (fonts), TMLN (timelines), OBJT (objects), ROOM (rooms), DAFL (datafiles), TPAG (texture pages), CODE (bytecode), VARI (variables), FUNC (function table), STRG (string table), TXTR (texture data), and AUDO (audio data).

    TXTR stores PNG-compressed texture pages at runtime; each TXTR entry contains a length-prefixed PNG blob. The PICA200 cannot read PNG; texture data must be decompressed on the ARM11 before upload. Cinnamon replaces TXTR and AUDO entirely with pre-baked assets in romfs: swizzled .t3x texture atlases (produced by the tex3ds tool) and .bcwav audio files. The data.win parser in Cinnamon skips TXTR and AUDO chunks at load time and reads pre-baked equivalents from romfs at asset initialization.

    The CODE chunk holds GML bytecode. Each code entry has a name (referenced by the FUNC table) and a byte array of VM instructions. The bytecode is a stack-based ISA. Each instruction is a fixed 4 bytes: 1 byte opcode, 1 byte type specifier, and 2 bytes operand or offset. The instruction set covers arithmetic (add, sub, mul, div, mod), comparison (cmp, set), stack manipulation (push, pop, dup), control flow (jmp, jt, jf, ret, exit), variable access (pushvar, popvar), and function calls (call, callv). The WAD version number (GEN8 offset 0x48) determines which opcode encodings and bytecode features are available; versions 16 and 17 are what Cinnamon currently supports.

    GML Bytecode Execution Model

    The GML VM is a stack machine. The interpreter maintains an execution stack (operand stack), a call stack (return addresses and local variable frames), and the instance context (self, other, global). Each frame stores local variables as a flat array indexed by slot number from the VARI table. When GML calls a script, a new frame pushes onto the call stack; on return, the frame pops and the return value sits on top of the operand stack.

    The with() statement is the most expensive GML construct for an interpreter. with(obj) creates an iterator over all instances of that object type. For each instance, the interpreter: saves the current self context, sets self to the iterated instance, executes the code block, then restores the previous context. With 200 instances and a 10-instruction body, this produces 200 context switches, 200 * 10 opcode dispatches, and 400 stack frame saves/restores. On O3DS ARM11 with Cinnamon's optimized dispatch, a with() loop over 200 instances with 15 opcodes per body takes approximately 0.5 to 1.0 ms depending on L2 residency. DR Chapter 4's 8,383 with() calls in the codebase include many loops over 50 to 200 instances; the heaviest rooms sustain 5 to 15 ms of interpreter time per frame on O3DS.

    Cinnamon's interpreter uses a computed goto dispatch table (GCC extension) rather than a switch statement. This removes the branch predictor penalty (which does not exist on ARM11 anyway) but more importantly it improves code locality: each opcode handler is a small block adjacent in memory, and the dispatch overhead is a single indirect branch load. The code override system allows hot-path GML functions (draw_sprite, draw_text, instance_place, collision_rectangle) to be replaced at load time with hand-written C implementations. The runner detects these by function name hash from the FUNC table and patches the call instruction to jump to the C version. This is the "code override" mechanism referenced throughout this document.

    GameMaker Texture Groups and Batch Breaks

    GameMaker organizes sprites into texture groups in the IDE. At build time, all sprites assigned to a group are packed into one or more power-of-two texture pages. At runtime, drawing from two different texture pages in sequence causes a texture swap and a batch break; the GPU pipeline stalls briefly to bind the new texture. GameMaker's own batch system flushes the current vertex buffer on: shader change, texture page change, surface target change, blend mode change, draw_rectangle or other primitive draws, tilemap draws, and view changes. This list (documented by JujuAdams in a widely-cited gist) is what defines the per-frame batch count in any GameMaker game.

    On desktop hardware, a batch break costs tens of microseconds; on ARM11 at 80 MHz effective, it costs hundreds of microseconds because the CPU must submit the current vertex batch via the GX queue, validate state, and rebuild the vertex stream for the next batch. citro2d's per-sprite-draw model means every sprite is a potential batch break if it differs from the previous sprite's texture page. With 200 sprites per frame from 50 different pages (typical for a complex Deltarune room before atlasing), citro2d produces up to 50 batch submissions per frame. citro3d's manual batching reduces this to 10 to 30 submissions by sorting by texture page and merging all sprites sharing a page into one draw call.

    GameMaker's sprite_add() and other runtime-loaded assets each create a new texture page outside the IDE-managed texture group system. Drawing these causes texture swaps that bypass the texture group optimizer. Cinnamon never calls sprite_add(); all textures are pre-baked into atlas pages and loaded into FCRAM at room init. No runtime texture page creation occurs.

    GML surfaces and Why They Are Expensive

    In GameMaker, surface_create() allocates a render target. All subsequent draw calls between surface_set_target() and surface_reset_target() write to that surface instead of the screen. Surfaces live in GPU texture memory (VRAM on desktop; VRAM on 3DS). On desktop, surfaces are cheap: GPU memory is fast and abundant, and switching render targets is a GPU command. On 3DS, surfaces are expensive for three reasons: they consume VRAM from the 6 MB pool (384 KB each at RGBA8 full resolution), switching render targets requires a PDC display transfer (DMA from VRAM to framebuffer), and the ARM11 must submit the target-switch command to the GX queue. GameMaker also treats surfaces as volatile: they are destroyed when the application loses focus, when GPU memory runs out, or (on 3DS) when VRAM is needed for something else. surface_exists() checks must guard every surface use.

    Deltarune's chapter-by-chapter surface count progression (0 in Chapter 1, 25 in Chapter 2, 111 in Chapter 3, 129 in Chapter 4, over 150 in Chapter 5) directly tracks the progression of lighting, compositing, and post-process effects. Each chapter adds more composited dark world layers, camera surfaces, and effect surfaces. On citro2d, each active surface requires a permanent VRAM slot for the duration it remains active. Six simultaneous surfaces at RGBA8 full resolution consume 2.25 MB of the 6 MB VRAM, leaving 3.75 MB for atlas textures, framebuffers, and depth buffers. Chapters 4 and 5 exceed this budget on certain rooms.

    The citro3d render-then-copy approach resolves this: each surface renders into VRAM one at a time, then immediately copies to FCRAM (via C3D_RenderTargetSetOutput with a GX display transfer), freeing the VRAM slot. The simultaneous VRAM requirement drops from (number of active surfaces × surface size) to (1 surface slot × surface size) regardless of how many surfaces the game has created. This is the single biggest architectural difference between the citro2d reference codebase and the citro3d fork.

    GameMaker Extensions and Why Most Fail on 3DS

    GameMaker extensions are native code plugins. A GMS 1.4 extension is a Windows DLL; a GMS 2 extension can be a DLL (Windows), .dylib (macOS), .so (Linux/Android), or a static library for console targets. When GML calls an extension function, control leaves the VM entirely and enters the native library. The library is loaded by the runner at startup via LoadLibraryA (Windows) or dlopen (Linux).

    On 3DS, none of these loading mechanisms exist. The homebrew runtime is a bare ARM11 executable; there is no dynamic linker, no DLL loader, and no OS-provided plugin system. Any GML code that calls an extension function will reach Cinnamon's function dispatcher, find no implementation, and either error out or silently return zero depending on how Cinnamon handles unimplemented built-ins. FMOD (Pizza Tower), Steam SDK extensions (steam_init, steam_achievement_set, etc.), crash reporter extensions, analytics extensions, and any other native plugin will all fail this way. The correct solution for each extension category is either to stub all its GML-facing functions (returning success codes that satisfy the calling code's error checks) or to reimplement the functionality using native 3DS APIs where applicable (BCWAV for audio, no equivalent for Steam).

    Pizza Tower's FMOD integration is the highest-profile example: the game makes approximately 40 to 60 distinct FMOD event calls per room transition and per gameplay moment. Every call must be stubbed or replaced. The Switch port's approach (TurtleHouse, D-Official) of completely reimplementing the fmod_sound event system in GML using GameMaker's native audio engine (then adapted further for 3DS to use BCWAV output) is the reference implementation for this class of problem.

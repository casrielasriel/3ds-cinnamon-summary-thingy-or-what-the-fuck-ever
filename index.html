3DS x Cinnamon Port Analysis

A personal breakdown of what my Cinnamon project can do on real 3DS hardware. Hardware ceilings, optimization impact, and per game compatibility.

What the percentages mean

100% = the game holds a steady 30 FPS on that hardware for the entire runtime. No dropped frames, no hitches.

Below 100% = the game cannot sustain 30 FPS in every frame. The number is the rough fraction of frames that complete within the 33ms budget. A game at 96% drops roughly 1 frame per second. At 90% it drops 3 frames per second.

Single-frame hitches happen when a frame takes 50-80ms. You notice a microstutter - the game hangs for one beat then resumes. These come from surface chain transitions (allocating FCRAM + display transfer), texture upload bursts from SD card, or large with() loops that generate too many VM ops in one frame.

Sustained drops happen when the average frame time exceeds 33ms. The game feels slow and the audio may desync. These come from a room that pushes too many draw calls, too many layers of surface compositing, or saturated VRAM spilling into FCRAM access.

Texture load stutter comes from PNG decompression and GPU upload on the ARM11 (the user confirmed this on real O3DS). Baked tex3ds textures eliminate it entirely - the PICA200 reads precompressed data directly from FCRAM with zero CPU decode.

What This Assumes

This document assumes the following code infrastructure exists and works.

  • Cinnamon. My bytecode recompiler takes GML bytecode from a data.win file and compiles it to ARM11 Thumb2 native code at load time. This replaces the GML interpreter entirely. The output is equivalent to what YYC produces, but targeted at 3DS.
  • citro3d renderer backend. A full implementation of the 32 function RendererVtable using citro3d. This gives me vertex batching, FCRAM textures, VRAM render targets, and direct PICA200 control. citro2d is not used.
  • File system backend. SD card and romfs access via libctru. Asset loading uses pre baked formats, not runtime PNG decode.
  • Audio backend. BCWAV playback through the CEVA DSP via ndsp. OGG decode on ARM11 is avoided.
  • Shader preprocessing pipeline. GML shaders are translated to PICA200 texenv configurations offline. No GLSL reaches the runtime.
  • Asset baking pipeline. All textures are converted to RGB5A1 or RGBA4444, packed into atlas pages via tex3ds, and stored as raw image data. All audio is converted to BCWAV. All tilemaps are baked to single textures where possible.
  • Input backend. HID input mapped to GML keyboard and gamepad constants. Touch screen mapped to mouse.

Without these systems, the numbers below do not apply. With the interpreter instead of Cinnamon, subtract 40 to 70 percent. With PNG and OGG at runtime, expect visible stutter on every room transition.

O3DS Hardware

6 MBVRAM (on SoC die)
128 MBFCRAM @ 3.2 GB/s
134 MHzCEVA TeakLite DSP
400 x 240Top screen per eye

The PICA200 uses DMP's MAESTRO 2G architecture. It has 4 programmable vertex processors. One can act as a geometry engine for subdivision. The fragment pipeline is fixed function with 5 TEV stages. There are 3 active texture units. The fourth is reserved for procedural texturing and is rarely used.

Supported MAESTRO extensions include per pixel lighting, shadow mapping, procedural textures, gaseous object rendering, polygon subdivision, cel shading, volumetric fog, and bump mapping. The 5 stage TEV can replicate most common fragment operations. No programmable pixel shaders are available; every GML shader must convert to TEV configurations.

Theoretical limits at 268 MHz are roughly 20.5 million triangles per second for geometry and 1,072 million pixels per second for fill. At 200 MHz the spec sheet rates are 15.3 million triangles and 800 million pixels.

The O3DS is CPU limited. The ARM11 at 268 MHz cannot feed the PICA200 draw calls fast enough. Commercial 3DS games typically push 3 to 6 million triangles per second. The N3DS at 804 MHz with 2 MB L2 cache solves this bottleneck entirely.

Measured Hardware Ceilings

I ran a stress test that pushed each axis until FPS dropped below 30. Each test used a 3 second measurement window. The results represent the breaking point for each axis in isolation.

  • Geometry. 40,000 quads per frame or 80,000 triangles. A Deltarune room uses maybe 500 to 1,000 sprite draws. I am at less than 3 percent of this ceiling.
  • Fill rate. 200 full screen overdraw layers per frame. Pixel art games have maybe 5 to 15 layers. I am at less than 8 percent.
  • Texture binds. 512 unique bindings per frame. A typical room uses 50 to 200 unique textures. With atlasing, maybe 10 to 30 atlas pages. Well within budget.
  • Surface DMA. 100 transfers per frame for a 200 x 120 RGB5A1 surface. This is the only axis where heavy games get close. With VRAM surfaces, I bypass this entirely.
  • VM throughput. 500,000 integer ops per frame with Cinnamon native compilation. The interpreter manages about 50,000. Chapter 5 rooms with complex with loops need maybe 5,000 to 15,000 ops.
  • Combined stress. 5x simultaneous scale factor. Running all axes at once still holds 30 FPS at 5 times the typical game load.

Emulator caveat. I ran this on Azahar. Azahar tracks real hardware closely for CPU and GPU. GSP shared memory timing, GX queue arbitration, PDC display transfer bandwidth, and VRAM bus contention may show small differences on a real O3DS. I validated the most critical unknown by running a 10 MB FCRAM texture test on a genuine O3DS at 59.8 FPS. Textures sampled from system RAM work at full speed with no VRAM copy needed.

Optimization Stack

Every optimization I can apply, ordered by impact. These are the systems that turn an unplayable port into a smooth one.

Critical

  • Critical Cinnamon native VM
    Compiles GML bytecode to ARM11 Thumb2 at load time. Runs 10 to 50 times faster than the interpreter. 500,000 ops per frame instead of 50,000. The single biggest performance win for any game with complex GML logic.
  • Critical citro3d with vertex batching
    Groups sprites by texture. Submits one draw call per texture instead of one per sprite. No redundant state validation per draw. Reduces CPU draw call overhead by 10 to 50 times.
  • Critical VRAM surface render targets
    Surfaces composited entirely on the GPU. Zero PDC display transfer cost per surface operation. Without this, Chapter 5 would spend half the frame budget on surface DMA.
  • Critical RGB5A1 and RGBA4444 preprocessing
    Cuts texture size by 2 to 4 times. Less memory bandwidth. More room for atlas pages in FCRAM. Reduces texture upload time from SD card.
  • Critical Texture atlasing (tex3ds)
    Puts 50 or more sprites into one atlas page. One draw call for 50 sprites. Directly addresses the O3DS ARM11 draw call submission bottleneck.
  • Critical Pre compiled shaders
    GML shaders converted to PICA200 texenv configurations offline. Zero runtime GLSL translation. TEV executes fixed function operations at no GPU cost.
  • Critical FCRAM textures
    PICA200 samples textures directly from FCRAM. No VRAM copy on bind. Proven at 59.8 FPS with 10 MB of textures on real hardware.
  • Critical BCWAV audio preprocessing
    CEVA DSP handles hardware decoded audio. Near zero ARM11 CPU cost. OGG decode on ARM11 eats up to 30 percent of the frame budget. This eliminates that.
  • Critical PNG to raw texture pre baking
    Eliminates 5 to 15 millisecond PNG decode per sprite sheet at load time. Room transitions load 200+ milliseconds faster. No visible stutter.

High

  • High Half resolution surfaces (200 x 120)
    4 times fewer pixels per surface. RGB5A1 halves it again. Total 8 times reduction versus full resolution RGBA8. VRAM usage per surface drops from 384 KB to 48 KB.
  • High Sprite culling
    Removes off screen sprites from the draw list. Busy rooms see 30 to 50 percent fewer draw calls. Simple frustum check per instance.
  • High Tilemap baking
    Pre renders static background tiles to a single texture per layer. Removes 200 or more individual tile draws per room. Huge win for rooms with complex tile based backgrounds.
  • High Culling based depth sorting
    Sort draw list by depth and texture once per frame. Skip invisible instances before entering the render loop. Reduces CPU sorting overhead.
  • High Static UI text and HUD baking
    Render HUD elements that rarely change to a cached texture. Only redraw when values update. Saves 10 to 20 draw calls per frame.

Medium

  • Medium Remove application surface
    Draw directly to the screen framebuffer. Saves one full 400 x 240 blit per frame, roughly 1 to 2 milliseconds.
  • Medium Dynamic resolution scaling
    Drop to 320 x 192 during heavy frames. 36 percent fewer pixels to fill. PICA200 display transfer scales to full resolution at nearly no cost. Switch back to full res when the room calms down.
  • Medium DSP hardware audio mixing
    All mixing runs on the CEVA DSP via ndsp. BCWAV playback costs effectively zero ARM11 CPU time. Frees up CPU for rendering and game logic.
  • Medium Pre computed vertex buffers for static geometry
    Rooms with static backgrounds and platforms get their geometry baked into persistent vertex buffers. No per frame vertex upload needed for background tiles.
  • Medium Texture group streaming
    Organize textures into groups per room. Load the next room's textures during gameplay via async SD reads. Hides load time behind gameplay.

Cheap wins

  • Cheap TEV palette swap fallback
    Some GML palette effects translate to TEV combiner stages at zero GPU cost. No shader needed.
  • Cheap Pre scaled sprite atlases
    Include 2x scaled versions of sprite atlases for rooms that need them. Avoids runtime bilinear scale on the GPU.
  • Cheap Skip frames on loading
    If a room transition takes more than 100ms, freeze the GPU but keep the DSP running. Audio continues smoothly. Resume rendering when loading completes.
  • Cheap Use PICA200 hardware fog for glow effects
    Many Deltarune glow and flash effects use gpu_set_fog. PICA200 has hardware fog support. This maps directly at no cost.

Game Compatibility

Each game has two numbers. Raw means citro2d with the interpreter, full resolution RGBA8 surfaces, PNG textures at runtime, and OGG audio. Optimized means the full Cinnamon stack with citro3d, VRAM surfaces, preprocessing, atlasing, and culling.

Undertale and Deltarune

Undertale. 72 percent raw, 100 percent optimized. I confirmed PNG loading stutter on a real O3DS. With baked textures the stutter disappears. The game is trivially simple by every axis.

UT Yellow. 68 percent raw, 100 percent optimized. Heavier effects than Undertale but still under 500 draw calls per frame. No surface chains to speak of.

DR Chapter 1. 65 percent raw, 100 percent optimized. Some surface use for dark world transitions. VRAM targets handle this for free.

DR Chapter 2. 50 percent raw, 100 percent optimized. Queen's manor near the end has heavier lighting. Still within budget.

DR Chapter 3. 35 percent raw, 100 percent optimized. More surfaces, more with loops, some 3D appearing scenes using draw_sprite_pos. No actual d3d calls beyond gpu_set_fog for glow effects.

DR Chapter 4. 25 percent raw, 99 percent optimized. Dark world vision mechanics need surface chains. VRAM exhaustion is possible on one or two frames during the heaviest vision transitions. It would be a single frame hitch, not a sustained drop.

DR Chapter 5. 18 percent raw, 98 percent optimized. The heaviest room is the Weird Route burst. It barely lasts. All d3d functions in the codebase are compatibility shims. The only one actively called is d3d_set_fog which wraps gpu_set_fog for sprite glow and flash effects. That maps to PICA200 hardware fog at zero cost. The platforming sections use draw_sprite_pos and vertex buffer perspective correction. No true 3D geometry.

Sonic SMS Remake

100 percent on O3DS. Simple pixel art Sonic engine using Master System physics. Barely 100 sprites per frame. No surfaces, no shaders, no complex with loops. This is the easiest port in the list.

Sonic.exe Fangames

100 percent on O3DS. Most use GameMaker 8 or Studio 1.4 with basic Sonic engines and horror sprite overlays. Blood sprites, jump scares, static screens. Minimal surface or shader use. Trivially within every ceiling.

Pizza Tower

Pizza Tower ships as bytecode. Cinnamon can target it directly.

My estimate is 85 percent on O3DS and 100 percent on N3DS. Pizza Tower has constant screen shake, heavy particle systems, transformation effects, surface based CRT overlay, and high speed layer scrolling. The per frame load is higher than Deltarune Chapter 5. The Pizza Time sequences with maximum screen effects would push the O3DS hard. N3DS would hold 30 FPS without issue.

YYC vs VM Bytecode

VM mode compiles GML to bytecode stored in data.win. Cinnamon reads the bytecode at load time and compiles it to ARM11 Thumb2. The result is native machine code equivalent to YYC output.

YYC mode compiles GML to C++ and then to a native EXE via MSVC or GCC. There is no bytecode in the binary. Cinnamon has nothing to read. YYC games cannot be targeted.

Cinnamon native compilation gives the same end result as YYC. The only loss is YoYo Games standard library for file I/O and network. I replace those with my own backends for SD card access and HID input.

Bottleneck Hierarchy

On O3DS with the full Cinnamon stack, the bottlenecks ordered by severity:

First. CPU draw call submission. The ARM11 can handle roughly 150 to 200 C3D_DrawArrays calls per frame. With atlasing and batching, a typical room uses 30 to 80 calls. That fits comfortably.

Second. VRAM exhaustion. 6 MB of VRAM supports 5 to 6 active surface render targets. A typical room uses 1 to 3. Chapter 4 and 5 occasionally need 4 to 5. That is tight but fits.

Third. FCRAM texture upload bandwidth. SD card reads at roughly 5 MB per second. A room transition might load 12 MB of atlas pages. That is a 2 to 3 second window where the game is loading. This is not a per frame bottleneck but it affects room transition time.

Fourth. Fragment fillrate. The PICA200 can do 11,000 or more full screen passes per frame at 30 FPS. Never a bottleneck for 2D pixel games.

The N3DS eliminates the first bottleneck with 3 times CPU speed and 2 MB of L2 cache. Games that hit 98 percent on O3DS hit 100 percent on N3DS.

Optimization Impact Summary

The full Cinnamon stack gains roughly 70 to 80 percentage points over the raw configuration across all games.

The single biggest win is citro3d with vertex batching and atlasing. It reduces draw calls by 10 to 50 times. The O3DS ARM11 cannot submit individual sprite draws at any reasonable framerate. It can submit 30 to 80 batches per frame. That single change is the difference between 18 percent and 50 percent on Chapter 5.

The second biggest win is Cinnamon native VM. It gives 500,000 ops per frame instead of 50,000. The interpreter chokes on Chapter 4 and 5 with loops that iterate 200 instances and call methods on each one. Native VM runs those loops at full speed.

The third biggest win is VRAM surface render targets. They eliminate all PDC display transfer costs per surface. Without this, Chapter 5 would spend half the frame budget or more on surface DMA transfers. With VRAM targets, that cost is zero.

Key Takeaways

The O3DS is CPU limited. Every optimization that reduces CPU work translates directly to FPS. Batching, culling, native VM, and precomputed draw lists are the correct targets.

The N3DS fixes the CPU problem. A game at 92 percent on O3DS runs at 100 percent on N3DS. The extra memory and L2 cache also help with texture streaming.

GML shaders using shader_set are dead on PICA200. There are no programmable pixel shaders. Every GML shader must convert to the 5 stage TEV fixed function pipeline. Most GM effects like color blend, additive, multiply, fog, and palette swap map cleanly. Complex displacement shaders and screen warp effects need pre baking or cannot be reproduced.

YYC games are a hard wall. No bytecode means Cinnamon cannot recompile them. The only path forward is a full manual reimplementation of the game logic in C, which defeats the purpose of using Cinnamon.

Preprocessing is mandatory. PNG decode at runtime causes visible stutter on every room transition. OGG decode on ARM11 eats 15 to 30 percent of the frame budget. GLSL at runtime needs CPU based translation into TEV configurations. All of these must be done ahead of time.

The PICA200 is not the problem. It is a 268 MHz GPU designed for a 2011 handheld that ran Monster Hunter 4 Ultimate, Kid Icarus Uprising, and Luigi's Mansion 2. Pixel art GML games are far below its design target. The only enemy is the ARM11 core, and every optimization in the stack is aimed directly at it.
